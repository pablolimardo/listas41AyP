El ESP32 tiene WiFi y el código no da error. Pero el broker no recibe nada.

¿Cuáles son las tres cosas que tenés que verificar ANTES de tocar el código?

💡
La respuesta no está en el código. Está en la red.
Sin IP no hay red. Sin gateway no hay salida. Sin DNS no hay nombres.
El código puede estar perfecto y el sistema no funcionar por ninguna de estas tres cosas.

// 01 · Los cinco conceptos

Lo que tenés que saber sí o sí

Sin estos cinco conceptos no se puede diagnosticar ningún problema de red.

🏠
① Dirección IP
El número que identifica a cada dispositivo en la red
📪
Es como el número de casa en una manzana. Sin número, el cartero no sabe dónde entregar. Sin IP, los paquetes de datos no saben dónde ir.
ESP32 192.168.1.45 Laptop/Node-RED 192.168.1.10 Broker Mosquito 192.168.1.100 Gateway (router)192.168.1.1 # Regla: en la misma red todas las IPs empiezan igual # No puede haber dos dispositivos con la misma IP
🧲
② Máscara de red
Define quién está en la misma red y quién no
🔌
Le dice al dispositivo quiénes son sus vecinos de manzana (misma red) y quiénes son de otra manzana (otra red).
Mascara: 255.255.255.0 192.168.1.45 -> red: 192.168.1 -> MISMA RED 192.168.1.100 -> red: 192.168.1 -> MISMA RED 192.168.2.10 -> red: 192.168.2 -> OTRA RED # El ESP32 en .1.45 llega directo al broker en .1.100 # Para llegar a .2.10 necesita pasar por el gateway
🚪
③ Gateway — puerta de salida
El equipo que conecta tu red local con el resto
🚫
Si querés salir de tu red (llegar a internet o a otra red), todo pasa por el gateway. Sin gateway solo podés hablar con los de tu misma red.
Gateway tipico: 192.168.1.1 (el router o la ONT) [ESP32] -> [Gateway 192.168.1.1] -> [Internet / broker externo] # Si el broker esta en la misma red -> no necesitas gateway # Si el broker esta en internet -> gateway obligatorio
💡 Caso del proyecto: si el broker Mosquitto corre en la laptop y el ESP32 está en la misma red WiFi, el gateway no importa para comunicarse entre ellos.
🔎
④ DNS — traductor de nombres
Convierte nombres en IPs (broker.local -> 192.168.1.100)
📓
Los dispositivos hablan en números (IPs). Las personas hablan en nombres. El DNS hace la traducción. Si falla, solo funcionan las IPs directas.
# Con nombre de host -> NECESITA DNS client.connect("broker.hivemq.com", 1883) # el ESP32 pregunta al DNS: que IP es broker.hivemq.com? # si no hay DNS o falla -> no puede conectarse # Con IP directa -> NO necesita DNS client.connect("192.168.1.100", 1883) # va directo, sin preguntar a nadie
Clave para el jueves: si el ESP32 se conecta por nombre y falla, probá con la IP directa. Si funciona con IP pero no con nombre → es DNS, no la red.
⚙️
⑤ DHCP — configuración automática
Asigna IP, máscara, gateway y DNS al conectarse
🤖
Cuando el ESP32 se conecta al WiFi le pregunta al router: "¿qué IP me das?" El router responde con todo lo necesario de una sola vez.
# En MicroPython: ver la configuracion asignada print(sta.ifconfig()) # -> ('192.168.1.45', '255.255.255.0', '192.168.1.1', '8.8.8.8') # IP Mascara Gateway DNS # Si muestra ('0.0.0.0', ...) -> DHCP no asigno nada # El ESP32 no esta realmente conectado a la red

// 04 · Diagnóstico aplicado

Tres casos reales — encontrá el problema

Tapá cada caso, analizalo y después revelá el diagnóstico.

A
Todo falla desde el primer ping
C:\> ping 192.168.1.1 Request timeout for icmp_seq 0 Request timeout for icmp_seq 1 C:\> ping 8.8.8.8 Request timeout C:\> ping google.com Request timeout
🔍 Problema LOCAL — Capa 1 o Capa 3

El dispositivo no llega ni al gateway. El problema está antes de salir de la red.

Verificar: ¿Tiene IP asignada? ¿Está conectado al WiFi? ¿DHCP activo?

En el proyecto: sta.ifconfig() devuelve 0.0.0.0 → WiFi conectado pero sin IP. Verificar contraseña o reiniciar el router.
B
Llego al gateway pero no a internet
C:\> ping 192.168.1.1 Reply from 192.168.1.1: time=1ms OK C:\> ping 8.8.8.8 Request timeout for icmp_seq 0 C:\> ping google.com Request timeout
🔍 Problema de SALIDA — Capa 3 (router/ISP)

La red local está perfecta. El problema está después del gateway.

Verificar: ¿El router tiene conexión WAN? ¿La ONT de Cotecal tiene luz?

En el proyecto: el ESP32 llega al broker local sin problema. Si el broker está en la misma red, funciona igual.
C
La red funciona pero los nombres no resuelven
C:\> ping 192.168.1.1 Reply from 192.168.1.1: time=1ms OK C:\> ping 8.8.8.8 Reply from 8.8.8.8: time=12ms OK C:\> ping google.com Ping request could not find host google.com
🔍 Problema de DNS — Capa 7

La red funciona perfectamente. Solo falla la traducción de nombres.

Solución: configurar DNS manualmente: 8.8.8.8 (Google) o 1.1.1.1 (Cloudflare).

En el proyecto: client.connect("broker.hivemq.com") falla pero client.connect("34.200.75.22") funciona → es DNS. Usar IP directa o corregir el DNS.
El Caso C es el más frecuente y el más confuso: parece que "la red no funciona" cuando en realidad funciona perfecto. La diferencia entre ping a IP y ping a nombre lo revela de inmediato.
